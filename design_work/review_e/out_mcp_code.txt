### 维度 1：协议正确性

**[major] mcp_server.py:352 — `_tool_result_content(payload)` 在 try/except 之外，序列化失败会冒泡为 -32603**

依据：`_handle_tools_call` 的 try/except（344-351）只包裹 `_TOOL_IMPL[name](store, args)` 的调用。第 352 行 `_tool_result_content(payload, is_error=False)` 在 try 块外。若 payload 含不可 JSON 序列化值，`json.dumps`（第 300 行）抛异常 → 穿透 `_dispatch` → 被第 427 行兜底捕获 → 回 -32603。违反「工具层任何失败绝不冒泡为 -32603」的绝对契约。

建议：将第 352 行纳入 try 块，或把整个 `_handle_tools_call` 函数体包在 try/except 中统一归一为 isError。

---

**[major] mcp_server.py:96 — `from .models import GraphStore` 在 try/except 之外，ImportError 冒泡为 -32603**

依据：`_StoreHolder.get()` 第 95 行先置 `_loaded = True`，第 96 行做 lazy import。若该 import 抛 ImportError（如 models.py 损坏），异常穿透 `holder.get()`（调用点 339 行无 try）→ `_dispatch` → 427 行 -32603。同时 `_loaded=True` 已置而 `_error=None`，后续调用返回 `(None, None)` → graph_unavailable 的 message 为 null，不可读。

建议：将 96-111 行整体包入 try/except，异常时写 `self._error` 后 return；或将 `_loaded = True` 移至成功路径末尾。

---

**[nit] mcp_server.py:55 — `_INVALID_PARAMS = -32602` 定义后从未使用**

依据：全文无任何引用。死代码。

建议：删除，或在工具入参校验处考虑是否该用 -32602（但按当前契约工具入参错走 isError，不用协议层错误码，故删除即可）。

### 维度 2：工具 schema 与描述准确性

**[nit] mcp_server.py:140/175/189 — `additionalProperties: false` 声明但服务端未强制校验**

依据：第 329 行 `args = params.get("arguments") or {}` 后，实现只取已知键（`args.get("question")` 等），对 schema 外的多余键静默忽略。schema 承诺 `additionalProperties: false` 但名实不完全相符（非 MCP 客户端的调用方可传多余键而不被拒绝）。

建议：在 `_handle_tools_call` 中按 `_TOOLS` 的 properties 键集做多余键检测，多余时返回 isError（`invalid_arguments`）；或在 description 中注明「多余键被忽略」以消除歧义。严格程度视项目取舍，nit 级。

### 维度 3：并发与超时

**[major]（与维度 1 第 2 条同根）mcp_server.py:95 — `_loaded=True` 先置于 import/load 之前，异常路径状态不一致**

依据：若 96 行 import 或 106 行 load 之前的代码抛异常，`_loaded` 已为 True 而 `_error` 为 None。后续 `get()` 直接返回 `(None, None)`，工具回传 `graph_unavailable` 但 message=null，丧失可读诊断。失败后不重试本身可接受（单进程、文件不变），但错误状态须正确记录。

建议：将 `_loaded = True` 移至函数末尾（成功与失败路径均经过后再置），或在 except 中确保 `_error` 被赋值。

---

**[nit] test_mcp_server.py:129-130 — `close()` 无 kill 兜底**

依据：`terminate()` + `wait(timeout=5)` 若超时（Unix 下进程忽略 SIGTERM），`wait` 抛 TimeoutExpired 被外层 except 吞掉，进程残留。Windows 下 `terminate()` 即 `TerminateProcess` 立即生效，实际风险极低。

建议：except 中补 `self.proc.kill()` + 短 wait。

- stderr 泵（测试 76 行）持续读取，服务器启动仅写 1-2 行 stderr（407-410），管道不会满阻塞。
- 测试客户端 queue + 后台读线程 + 30s 超时 + atexit 注册 + close 幂等（poll 前置检查），无竞态/僵尸风险。
