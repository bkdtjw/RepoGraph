## 1. 三件基础事实定稿（计划书 §0.2 留白项）

### 1.1 存储结构（GraphStore 内存结构 + graph.json 格式）

**内存结构**（`src/repograph/models.py:176-216`，`class GraphStore`）——两张字典，无索引、无邻接表预建：

- `self._nodes: dict[str, dict]`（`models.py:180`）：`node_id → {"id":…, "label":…, **properties}`。属性**内联**在节点字典里（非嵌套）。`merge_node`（`:183`）以 `node_id` 为合并键，`setdefault` + `update`，重跑幂等（§3.4）。
- `self._edges: dict[tuple[str,str,str], dict]`（`models.py:181`）：`(src, etype, dst) → properties`。`merge_edge`（`:189`）以三元组为键。
- 遍历靠全量扫描：`nodes(label)`（`:195`）、`edges(etype)`（`:200`）线性过滤；`counts()`（`:208`）现算 by-label / by-type 计数。检索层每次调用 `store.nodes()` 都是 O(N) 全表扫（`context.link_entities` / `impact._resolve_symbol` 均如此）。

**graph.json 磁盘格式**（`models.py:220-231` `save`，实测确认）——顶层两键 `{"nodes":[...], "edges":[...]}`：

- `nodes`：节点字典**原样平铺**成 list（id/label 与属性同级）。实测 Function 节点键：`id, label, repo, qualname, file, span_start, span_end, signature, is_async, is_method, is_endpoint, http_method, route_path, docstring`。
- `edges`：**结构与节点不对称**——每条边序列化为 `{"src":…, "type":…, "dst":…, "properties":{…}}`，属性收在嵌套 `properties` 键下（`models.py:225-228`）。`load`（`:233-244`）按此格式还原：节点剥离 id/label 后其余作 props，边取 `properties`。
- 写盘参数：`json.dump(..., ensure_ascii=False, indent=1)`（`:231`）。实测 510 节点 / 1698 边、文件约 724 KB。

**关键 as-built 事实**：Function 节点**无任何谓词指标属性**（无 `fan_in/pagerank/heat/cyclomatic/blast_radius/fix_involvement/churn`）、**无 `loc`**（`loc` 只在 Module 节点，`build.py:122` / `models.ModuleFacts.loc:131`）、**无 `zh_aliases`**。实测节点键清单即上列 14 键，印证 gap-matrix「谓词指标属性全无」「Function 无中文别名」。Concept 节点有 `aliases` 字段但 139 概念中近乎全空。

### 1.2 持久化方式（全量写盘？何时写？）

- **全量快照，无增量**：`GraphStore.save`（`models.py:220-231`）一次性 dump 全部节点+边，无 append、无 diff、无部分落盘。
- **只在离线 CLI 阶段写盘**，运行时只读：
  - `index` 阶段末 `store.save(graph_path)`（`cli.py:136`，`_run_index`）；
  - `semantic` 阶段末就地回写**同一** `graph.json`（`cli.py:157`，`_run_semantic`；语义层把 Concept 节点与 DESCRIBES/IMPLEMENTS 边 merge 进 store 后整体重写）；
  - 运行时（`claude-ui/server.py`）**从不 save 图谱**——只 `GraphStore.load`（`server.py:306`）。全文件无运行时 `store.save`。
- **运行时唯一写盘是会话状态**，与图谱分离：session JSON（`data/sessions/{sid}.json`，`server.read_session`/`write_session`，atomic_write），承载对话历史，未来 `rg_focus` 焦点栈拟挂此处（尚未落地）。

### 1.3 索引触发方式（手动 CLI 全量 build？有无增量？）

- **手动 CLI、全量重建**：`repograph index --repo <PATH> --name <NAME>`（`cli.py:284-288`），`_run_index`（`cli.py:102-144`）五步确定性全量流水线：AST 结构抽取 → 端点识别 → 调用图两遍扫描 → git 历史抽取 → `build.build_graph` 组装 → `save`。或 `repograph all`（`cli.py:314`）一键 `index → semantic → viz`。
- **无增量水位**：每次 `index` 都 `extract_worktree` 全量重解析整个工作树（`cli.py:114`），无 `(path,mtime)` 增量、无 diff 级更新。gap-matrix §0 增量索引水位记「未实现」，核实无误。
- **运行时是「整体热失效重载」，非增量更新**：`server._rg_load`（`server.py:286-328`）按 `graph.json` 的 `(path, mtime)` 判失效（`:300-305`），一旦变化就整体 `GraphStore.load` 换新 store，并连带重建 BM25 语料索引与概念名白名单（`server.py:316-317`）。即「换图谱 = 整体 reload」，图谱本身仍由离线 CLI 全量产出。

## 2. 语义通道现状（修正计划书 §0.2 过时表述）

计划书 §0.2 述「语义生成走 grok API」，此表述已过时。as-built 分两条独立通道：

- **grok 断供**：2026-07-22 grok CLI 订阅到期，402 Payment Required（落地设计文档头「通道变更」注记，实测确认；本盘点不复跑 402 请求）。
- **运行时 L2 语义 = 网关 flash（已在）**：`server._rg_llm_select_concepts`（`server.py:430-481`）向网关 POST `{gateway_base}/v1/messages`（`:460`），模型取 `cfg["model_haiku"]`（`:438`，flash 档），职责是「给全概念名清单 → 选相关概念名」（非生成）。程序侧白名单校验只采信清单内真实概念（`:472-481`），静默降级、绝不回显 token（`:462-468`）。**此路径本就不依赖 grok**，grok 断供零影响。
- **索引期语义 = grok CLI（已断供，待迁网关）**：`extract/semantic.py:30` `from repograph.extract.grok_client import ask_grok`，`semantic.py:184` `ask_grok(prompt, json_schema=CONCEPT_SCHEMA)`；`grok_client.py` 是 `grok -p` 的 subprocess 封装。grok 402 后此路径失败——但 `cli.py:264` 的 `all` 命令对 semantic 失败降级为告警，结构层产出不受阻。计划的索引期网关 HTTP 客户端 `extract/llm_client.py`（qwen3.8-max-preview）**尚未建**（`extract/` 目录仅 `grok_client.py` + `semantic.py`，无 `llm_client.py`）——待 Phase C 落地。
- **§0.2 修正稿建议**：「运行时 L2 语义走网关 flash（`_rg_llm_select_concepts`，已在）；索引期语义原走 grok CLI，2026-07-22 断供后待迁网关 HTTP（`extract/llm_client.py`，Phase C 建）。」
