# RepoGraph 补充设计:模糊语义处理(Query Understanding 层)

**版本**:v0.2 补充稿(附属于主文档 v0.1,引用其章节号)
**日期**:2026-07-22
**触发背景**:实测中 "你知道我的代码库吗" 类问题返回 "不知道"。根因不是生成模型能力,而是主文档 §7 的检索层缺少查询理解(Query Understanding)子层:无答案来源(§7.6 被砍)、无显式路由、无回退阶梯。本文档定义该子层的完整设计。

---

## 1. 问题定义

### 1.1 失效复盘(根因链)

以 "你知道我的代码库吗" 为例,在 v0.1 流水线中的完整路径:

```text
问题 → §7.1 实体链接:无任何符号词面 → 无锚点
     → 退化纯向量 RAG:chunk 库(docstring/commit/README 段)中
       不存在语义上"回答该问题"的文本 → top-8 全为噪声
     → 生成层执行"无法从上下文得出时明确说未覆盖、禁止编造"
     → 输出"不知道"    ← 系统在诚实地执行规范,而非理解失败
```

结论:**模糊语义问题的九成修复发生在索引期与路由层,而非生成 prompt**。给生成器加一句"请理解模糊表达"不改变任何一环的输入输出。

### 1.2 术语界定:三种不同的现象,三种不同的处理

严格区分,后文按此分流,混用会导致错配的解法:

| 现象 | 定义 | 例 | 正确处理 |
|---|---|---|---|
| **歧义(ambiguity)** | 离散的多个合法解读 | "run 这个函数"命中 9 个候选 | 消歧:反问或自选并披露(§5.5) |
| **模糊(vagueness)** | 谓词无明确边界,连续统上的程度词 | "核心模块""热点函数" | 操作化:映射为可度量的图代理并披露定义(§4.3) |
| **欠指定(underspecification)** | 信息不足以定位意图或指称 | "你知道我的代码库吗""帮我看看那块" | 补全:概览层作答 / 会话焦点栈 / 改写扩展 / 澄清(§4.1、§5.3、§5.6) |

---

## 2. 模糊语义分类学

八类现象。每类给出:检测信号(供 §5.2 路由器使用)→ 处理策略 → 归属机制。这张表是本设计的骨架,评测集(§8)按它逐类覆盖。

| # | 类别 | 例句 | 检测信号 | 处理策略 | 机制 |
|---|---|---|---|---|---|
| C1 | 元问题/能力问题 | "你知道我的代码库吗""你能干什么" | 元问题模式正则(附录 A) | 不检索;仓库卡片 + 能力清单直接作答 | §4.1 + meta 路由 |
| C2 | 全局概括 | "这项目是干嘛的""整体架构什么样""代码质量怎么样" | 全局关键词 + 无代码词元 + 链接无命中 | level-0/1 概览 + Concept Top-N + 结构统计 | §4.1 + global 路由 |
| C3 | 口语化指称(跨语言) | "上下文那块是怎么搞的""安全那一套" | 有内容意图、无可链接词面 | 双语改写扩展 → 二次链接 → 双语实体卡片召回 | §5.3 + §4.2 |
| C4 | 指代与省略(跨轮) | "它的调用方有哪些""那删除呢" | 指代词(它/这个/该/上面)且无其他锚点 | 会话焦点栈消解,结果回显披露 | §5.6 |
| C5 | 词面歧义 | "handler 在哪定义的"(12 个候选) | 链接返回多候选且分差 < δ | 消歧协议:候选列表返回 / 唯一强候选自选并披露 | §5.5 |
| C6 | 模糊谓词 | "最重要的函数""改起来最危险的地方" | 程度词/评价词词表命中 | 操作化为预计算指标,答案强制披露代理定义 | §4.3 |
| C7 | 错误预设 | "为什么 X 用了 Redis 分布式锁"(实际没用) | 改写阶段抽出的前提在图/块中无支撑 | 预设校验:指出并纠正,禁止顺着编 | §5.7 |
| C8 | 界外问题 | "Python 的 GIL 是什么" | 无仓库指称 + 通用知识模式 | 声明界外;是否顺带常识作答由配置决定 | out_of_scope 路由 |

C1/C2 的答案来源必须在**索引期**制造出来(v0.1 的直接缺口);C3–C5 是**查询期**的链接与消歧问题;C6/C7 横跨索引期(指标预计算)与生成期(披露与纠正);C8 是纯路由问题。

## 3. 总体方案与设计原则

三个介入面:**索引期**(§4,制造答案来源与双语可达性)、**查询期**(§5,理解、路由、链接、消歧、回退)、**生成期**(§6,披露与诚实约束)。五条原则,评审与代码评审均以此为据:

- **P1 答案来源前置**:凡是希望系统能回答的问题类型,其答案素材必须在索引期被显式制造(概览、指标、双语卡片),不指望检索"碰巧"覆盖。
- **P2 路由显式化**:v0.1 的隐式单分支(链接是否命中)升级为显式多路,路由决策可日志、可评测。
- **P3 确定性工具不吃模糊输入**:`impact_analysis` 等模板工具的入参必须是已消解的规范 ID;模糊到规范的转换只发生在链接层,且转换结果对调用方可见。
- **P4 永不裸拒**:任何路径的响应必须含至少一个可行动元素(答案 / 候选列表 / 概览 + 建议问法),"不知道"只能以"我的边界是什么 + 你可以怎么问"的形态出现。
- **P5 解释回显**:响应携带 `resolved_query`(系统对问题的最终解读,含指代消解与消歧选择),让调用方 Agent 与人类都能发现并纠正误解——误解可纠正,沉默的误解不可纠正。

## 4. 索引期增强

### 4.1 概览层(level-0 / level-1)

即 GraphRAG 社区摘要思想的最小实现:整个仓库是第 0 层社区,顶层包是第 1 层,不引入 Leiden。

**level-0 仓库卡片**(每仓库一张,JSON,注入相关路由的 system prompt,不走检索):

```json
{
  "repo": "agent-studio",
  "summary": "自托管 AI Agent 运行时:FastAPI + React,ReAct/Plan&Execute 双模式……(LLM 生成,150-300 字)",
  "stats": {"modules": 0, "classes": 0, "functions": 0, "commits": 0,
            "issues": 0, "concepts": 0, "time_span": "", "call_resolved_rate": 0.0},
  "entrypoints": ["…is_endpoint 节点聚合…"],
  "top_concepts": ["…confidence 加权 Top-5…"],
  "hot_functions": ["…§4.3 heat 指标 Top-5…"],
  "generated_at": "", "at_commit": ""
}
```

`stats/entrypoints/hot_functions` 全部确定性聚合;仅 `summary` 为一次 LLM 调用(输入 = 确定性字段 + README 首段 + Top 概念,输出限 300 字,quote 类校验不适用,但禁止出现未在输入中出现的专有名词——程序校验名词表)。

**level-1 包级概览**:每个顶层包(`app/core`、`app/api` …)一段 120 字摘要,存为 `source_type='overview'` 的 chunk 进向量层,同时挂为图上 `Module` 聚合节点属性。C2 类问题的检索空间 = level-0 卡片 + level-1 块 + Concept 节点,不进普通 chunk 池竞争。

**刷新策略**:随增量索引水位(主文档 §6.5)重算确定性字段;`summary` 仅在结构统计变化超阈值(模块数 ±10% 或新增顶层包)时重生成,避免每次增量都烧一次 LLM。

### 4.2 双语实体卡片与中文别名

C3 类失败的一半原因是跨语言词面断裂:问题是中文口语,标识符是英文。三个索引期动作:

1. **实体卡片扩充**:v0.1 的实体卡片块(`qualname + docstring 首行`)追加一句 LLM 生成的中文功能描述(≤40 字,输入为签名 + docstring + 所在模块概览;抽检 30 条,主观可信率 ≥85% 方可全量)。embedding 空间从此中英双语可达。
2. **中文别名入表**:核心模块、关键类与全部 Concept 的中文名/俗称批量写入 `symbol_alias`(`kind='zh_alias'`),来源 = 概念对齐产物 + 上述中文描述中的名词短语抽取。
3. **缩写扩展表**:配置化词表(`ctx→context, cfg→config, auth→authentication, db→database, msg→message …`),链接前对查询词元做双向扩展。

### 4.3 模糊谓词的操作化(指标预计算)

C6 类问题不可能有"客观"答案,正确做法是把模糊谓词映射为**公开定义的图代理**,并在答案中强制披露定义。指标全部索引期算好写入节点属性,查询期零成本:

| 触发词(词表配置化) | 代理指标 | 定义 |
|---|---|---|
| 核心 / 重要 / 关键 | `fan_in`、`pagerank` | CALLS 入度;CALLS∪IMPORTS 反向图 PageRank(networkx,索引期一次) |
| 热点 / 改得最多 | `heat` | `commits_all + 2 × commits_90d`(权重可配) |
| 复杂 / 难懂 | `cyclomatic`、`loc` | 近似圈复杂度 = 1 + 分支节点数(If/For/While/Try/BoolOp/推导式条件,AST 遍历顺带产出,P2 补入 §4.1 抽取器) |
| 危险 / 改起来风险大 | `blast_radius`、`fix_involvement` | 反向 CALLS ≤3 跳闭包大小 × 是否可达端点;历史上被 FIXES 提交修改的次数 |
| 没人用 / 死代码 | — | 主文档附录 A7 查询,非属性 |
| 不稳定 / 老在改 | `churn_90d` | 近 90 天 MODIFIES 计数 |

**披露话术模板**(生成层强制):"按 <代理定义> 排序,前 N 为 …;'<用户用词>' 在本系统中即按此定义度量。" 代理是定义而非真理,这句话把定义权交还给用户。

### 4.4 词元与停用处理

链接层 n-gram 提取增加中文停用与疑问词表(怎么/哪个/那块/搞的…),避免其进入词面匹配;代码词元检测器(camelCase / snake_case / 点路径 / 文件后缀 / `#数字` / 路由样式)同时服务于路由器(§5.2)与链接器。

## 5. 查询期流水线

七个阶段。稳态开销:规则路由命中时新增 0 次 LLM 调用;需要改写/兜底分类时新增 1 次廉价调用(改写与前提抽取合并在同一次,§5.3/§5.7)。

```text
问题 → S0 规范化 → S1 路由 ─┬─ meta ────────→ level-0 卡片作答(零检索)
                            ├─ global ──────→ 概览层检索(4.1)+ 统计
                            ├─ entity_local → S2 改写扩展 → S3 链接 v2 → S4 消歧
                            │                → local search(主文档 §7.4)
                            ├─ structural ──→ text2cypher / 模板(主文档 §7.2/7.3)
                            └─ out_of_scope → 界外声明
任意路径失败 → S6 回退阶梯(禁止裸拒)   全程:S5 焦点栈按需介入,S7 前提校验挂在生成前
```

### 5.1 S0 规范化

全半角统一、大小写保留(标识符大小写有信息量)、去首尾噪声、反引号内容标记为强代码词元。

### 5.2 S1 路由器

**规则优先**(零成本、可解释,初始规则表见附录 A),按序匹配,首中即出;全不中则 **LLM 兜底分类**(固定标签集 `{meta, global, entity_local, structural, out_of_scope}`,温度 0,输出 ≤30 token,按规范化问题缓存)。

**置信度与误路由代价不对称**:兜底分类置信度 <0.6 时按 `global` 处理——误入 global 的代价是精度损失但答案安全(概览总是相关的),误入 entity_local 的代价是拿噪声上下文生成,后者更糟。路由决策(命中规则 ID 或兜底标签 + 置信度)写入结构化日志,是 §8 路由准确率指标的数据源。

### 5.3 S2 改写与扩展(仅 entity_local 且直接链接无命中时)

一次 LLM 调用,同时产出三样(合并调用,控制延迟):

```json
{
  "queries": ["context compression resumable", "上下文 压缩 续传", "ContextManager"],
  "symbol_guesses": ["ContextManager", "compress", "context"],
  "premises": [{"claim": "系统使用 Redis 分布式锁", "entities": ["Redis"]}]
}
```

约束:`queries` 限 2–4 条、必须中英各至少一条;`symbol_guesses` 只作链接候选、**永不出现在最终答案的事实陈述中**(改写产物只进检索通道,防污染);可选启用 HyDE 变体(生成一段假设性答案文本做 embedding 检索),作为配置项默认关闭,评测集上 A/B 后再定。结果按规范化问题缓存(LRU 1k)。

### 5.4 S3 链接 v2(三段阈值带)

候选来源合并:别名精确/后缀(v0.1)∪ 缩写扩展命中 ∪ 双语实体卡片 embedding top-5(对原问题与全部改写查询分别检索后取并)。按最高分 s 分带:

- `s ≥ τ_hi`:自动锚定;
- `τ_lo ≤ s < τ_hi`:进入 §5.5 消歧协议;
- `s < τ_lo`:判定无锚点,走回退阶梯。

初始值 τ_hi=0.62、τ_lo=0.45、候选分差阈 δ=0.05——**这是待校准的初始值而非结论**,校准流程见 §8.4;词面精确命中恒视为 s=1。

### 5.5 S4 消歧协议(C5)

触发条件:分带落中段,或 Top-2 分差 < δ 且二者语义上均合法。行为分两支:

- **唯一强候选**(Top-1 领先 ≥δ 或其余候选同名但为测试代码/已弃用路径):自动选择,`resolved_query` 中披露("按 app/core 下的 ToolRunner.run 解读,另有 2 个同名候选已忽略")。
- **多合法候选**:不猜。响应置 `needs_disambiguation=true`,`candidates[]` 携带区分性元数据(路径、docstring 首行、fan_in),**由调用方 Agent 决定是否向人反问**——反问权留给持有对话上下文的一方,RepoGraph 只负责把候选摆清楚。

澄清预算:单次查询至多触发一次消歧,防止澄清循环。

### 5.6 S5 会话焦点栈(C4)

服务端进程内维护(stdio 模式一进程一会话,主文档 §8.1 假设下成立):最近 5 个成功锚定的实体,带轮次 TTL=10。仅当问题含显式指代词且无其他锚点时介入,取栈中最近的类型相容实体,消解结果必须回显进 `resolved_query`。定位是**纵深防御**:主消解责任在调用方 Agent(Claude Code 通常会把"它"展开后再调工具),本层兜漏网。`ask_repo` 同时新增可选参数 `context: list[str]`(调用方最近若干轮原文),提供时优先于焦点栈。

### 5.7 S7 前提校验(C7)

S2 抽出的 `premises` 逐条做轻量验证:实体可链接但图中无支撑边、且向量检索无高分支撑块 → 标记 `premise_unverified` 注入生成上下文。生成层规则(§6):被标记的前提必须在答案开头显式处理("图谱中未发现 X 使用 Redis 锁的证据;X 实际采用 …"),顺着未证实前提往下答按幻觉计(评测 PP 集单列该失败模式)。

### 5.8 S6 回退阶梯

```text
entity_local 检索空/低分 → 附加 level-1 概览 + embedding 最近实体 Top-3(标注 low-confidence)+ 建议问法
无锚点且意图仍像实体题   → 改写后二次链接 → 仍无 → 同上
out_of_scope             → 声明界外;answer_general=false(默认)仅声明,true 则顺带常识作答并标注非仓库知识
所有路径                 → 响应含 ≥1 个可行动元素;裸"不知道"在 §8 中作为硬失败指标(目标 0)
```

## 6. 生成期约束(prompt 增量)

在主文档 §7.4 生成规则之上追加四条,核心是把"诚实"从"要么答要么拒"细化为"答案 + 边界 + 出路"三件套:

1. **边界声明形态**:上下文不足时,禁止输出孤立的"不知道/未覆盖";必须输出三段式——已知的最接近信息(概览或近邻实体)、明确的边界("图谱未覆盖 X 的 Y 信息")、可行动建议(候选问法或工具)。
2. **代理披露**:凡答案依据 §4.3 指标,必须内联披露代理定义,不得把代理呈现为客观结论。
3. **消解回显**:凡发生指代消解、消歧自选、改写介入,`resolved_query` 必填且答案首句体现解读("按 ContextManager.compress 理解你说的'压缩那块'……")。
4. **前提处理**:`premise_unverified` 标记的前提必须显式纠正后再作答(§5.7)。

改写产物(`symbol_guesses`、HyDE 文本)在生成上下文中以 `[检索辅助,非事实]` 段落隔离,引用规则禁止引用该段。

## 7. 接口契约变更

### 7.1 ask_repo 响应 schema v2

```json
{
  "answer": "…",
  "mode": "meta | global | entity_local | structural | vector_only | out_of_scope",
  "resolved_query": "系统对问题的最终解读(含消歧/指代消解结果)",
  "anchors": [{"id": "…", "score": 0.71, "method": "alias | zh_alias | embedding | focus_stack"}],
  "needs_disambiguation": false,
  "candidates": [{"id": "…", "path": "…", "doc_head": "…", "fan_in": 12}],
  "premise_flags": [{"claim": "…", "status": "unverified"}],
  "citations": ["commit 9f3ab12", "…"],
  "degraded": false,
  "suggestions": ["可以问:修改 X 会波及哪些端点"]
}
```

v1 消费方(评测脚本)兼容:新增字段全部可选,`answer/citations/mode/degraded` 语义不变。

### 7.2 新增 MCP 工具 repo_overview

```python
@mcp.tool()
def repo_overview(repo: str | None = None) -> dict:
    """返回已索引仓库的 level-0 卡片(结构统计、入口、核心概念、热点)。
    适合'你了解这个项目吗/整体架构什么样'类问题;零检索、毫秒级。"""
```

C1/C2 类问题从此对调用方 Agent 直接可见可路由——工具描述本身就是路由提示。工具总数 3→4,仍然克制。

### 7.3 既有工具的模糊输入行为

`impact_analysis(symbol=...)` 收到非规范 ID 时走链接 v2:唯一强候选 → 执行并在响应回显 `resolved_symbol`;多候选 → 返回 `candidates[]` + `needs_disambiguation=true`,**不执行**遍历(P3 原则:确定性工具不吃未消解输入)。`query_graph` 不变(其输入本就是自然语言,text2cypher 内部消化)。

## 8. 评测扩展

### 8.1 新增四个子集(并入 eval/dataset.jsonl,type 字段区分)

| 集 | 数量 | 构造 | 判定方式 |
|---|---|---|---|
| L0 元问题 | 10 | C1 类模板变体(含口语/错别字变体) | 程序断言:mode=meta 且答案含卡片统计事实,无需 LLM 裁判 |
| FZ 口语指称 | 20 | 作者出题,**刻意与目标实体名零词面重合**,中文口语占半;每题标 gold_entity | anchor hit@1 / hit@3(程序判)+ 答案准确率(裁判) |
| AMB 歧义 | 10 | 每题标 gold_behavior ∈ {should_disambiguate, should_autopick} | 程序判:消歧触发与 gold_behavior 一致性 |
| PP 错误预设 | 8 | 含一条与图谱事实相悖的前提 | 裁判判"指出并纠正";顺着预设作答 = 幻觉,单列计数 |

出题纪律沿用主文档 §9.1:FZ 的 20 题拆 dev/test 各 10,dev 用于 §8.4 校准,test 冻结。

### 8.2 新增指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 路由准确率 | 全集上 mode 与人工标注一致率 | ≥ 0.9 |
| anchor hit@1 / hit@3 | FZ-test 上 gold_entity 命中 | 实测报告,dev 上优化 |
| 过问率 / 漏问率 | AMB 上 should_autopick 却反问 / should_disambiguate 却自选 | 过问率 ≤ 0.2,漏问率 ≤ 0.1 |
| 预设纠正率 / 预设幻觉率 | PP 集 | 纠正 ≥ 0.75,幻觉 = 0(硬指标) |
| 裸拒率 | 全集上无可行动元素的拒答 | 0(硬指标) |
| 澄清开销 | 平均每题消歧触发次数 | ≤ 0.15 |

### 8.3 主表扩展

主文档 §9.3 的四组对比矩阵增加一列 "v0.2(本设计)",与 v0.1 混合方案同题对比——本补丁的价值必须以 L0/FZ/AMB/PP 四个子集上的增量呈现,L1/L2/L3 允许持平(理论上不应下降,下降 >3pt 视为回归,阻断发布)。

### 8.4 阈值校准流程

FZ-dev 10 题上网格搜索 (τ_hi, τ_lo, δ):目标 max hit@1,约束 过问率 ≤0.2。冻结三元组后跑 FZ-test 与 AMB,校准集与测试集不得混用;校准结果(含网格全表)入 `eval/calibration.md`。

## 9. 成本与延迟预算

| 项 | 索引期一次性 | 查询期稳态 |
|---|---|---|
| level-0/1 概览 | ≤ 20 次 LLM 调用/仓库 | 0(注入 prompt) |
| 中文实体卡片 | ≈ 模块数 + 关键类数 次(数百次廉价调用) | 0 |
| 指标预计算(PageRank/圈复杂度等) | 秒级 CPU | 0 |
| 路由 | — | 规则命中 0;兜底 1 次(≤30 token,缓存) |
| 改写+前提抽取 | — | 直接链接命中 0;未命中 1 次(合并调用,缓存) |
| 端到端延迟增量 | — | 规则路径 +0ms;最坏路径 +1~2s(两次廉价调用) |

预算结论:稳态最多新增 1–2 次廉价 LLM 调用且均带缓存,不改变系统成本量级。

## 附录 A:路由规则初始表(配置文件 router_rules.yaml,按序首中即出)

```yaml
- id: meta-1
  label: meta
  pattern: '(你|您)?(知道|了解|认识|熟悉|清楚)\s*(我的)?(代码库|仓库|项目|repo|codebase)'
- id: meta-2
  label: meta
  pattern: '你(是谁|能(干|做)什么|有(什么|哪些)(功能|能力))|怎么用你'
- id: struct-1
  label: structural
  requires: [has_code_token]        # 代码词元检测器(§4.4)
  pattern: '(最|前\s*\d+|多少|几个|统计|列出(所有|全部)|排(序|名))'
- id: entity-1
  label: entity_local
  requires: [has_code_token]        # 出现标识符/路径/反引号/#issue 即实体题
- id: global-1
  label: global
  requires: [no_code_token, no_linker_hit]
  pattern: '(整体|总体|全局|大概|架构|介绍|讲讲|是(干|做)(什么|嘛)|质量|难点|亮点|风格)'
- id: oos-1
  label: out_of_scope
  requires: [no_repo_reference]
  pattern: '(是什么意思|什么是)\s*(?!.*(本项目|这个项目|仓库))'   # 通用概念问询
# 全部未命中 → LLM 兜底分类(§5.2)
```

规则表随评测迭代,每次增删必须附对应回归用例。
